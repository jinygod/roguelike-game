# 턴제 로그라이크 웹게임 프로토타입 설계

- 작성일: 2026-06-20
- 상태: 사용자 승인
- 범위: 챕터 1의 5개 스테이지, 플레이 로그 수집, 비공개 밸런스 분석 기반

## 1. 제품 목표

세 명의 캐릭터를 직접 지휘하고 증강 시너지를 쌓아 5개의 스테이지를 돌파하는 턴제 로그라이크 웹게임을 만든다.

첫 프로토타입의 성공 기준은 콘텐츠 양이 아니라 반복해서 플레이하고 싶은 하나의 챕터다. 챕터 1회 도전은 10~15분 안에 끝나며, 빌드가 완성될 때 피해와 효과가 연쇄적으로 폭발하는 재미를 제공한다.

재미의 우선순위는 다음과 같다.

- 증강 시너지와 빌드 폭발: 60%
- 적 행동을 읽고 피해를 줄이는 전술 퍼즐: 40%

## 2. 확정된 범위

### 포함

- 전사, 궁수, 법사로 구성된 고정 3인 파티
- 3개 공격로와 전·후방 위치를 사용하는 전투
- 챕터 1의 1-1부터 1-5까지
- 고블린 숲길 테마와 고블린 전차대장 보스
- 안전·위험 조건 선택
- 약 18개의 증강과 최소 4개의 빌드 방향
- 영구 콘텐츠 해금
- 익명 기기 ID 기반의 전체 행동 로그
- 비공개 밸런스 연구실의 데이터 구조와 단계적 구현
- Windows와 macOS 개발 환경
- PC와 모바일 가로 화면

### 제외

- 2~5챕터
- 회원가입과 사용자 계정
- 멀티플레이
- 공개 배포와 운영
- 공개 랭킹
- 상점, 장비 인벤토리, 복잡한 분기 지도
- 캐릭터 영구 능력치 강화

공개 배포는 로컬 플레이테스트와 밸런스 검증 후 진행한다. 다만 배포를 막는 운영체제 종속 코드나 로컬 절대 경로는 처음부터 만들지 않는다.

## 3. 플랫폼과 표현

게임은 React 기반 웹게임으로 구현한다.

- PC: 기본 전투 화면
- 모바일: 가로 모드 전투 화면
- 공통: 동일한 전투 규칙과 게임 상태
- 반응형 차이: 화면 배치, 터치 대상 크기, 정보 패널 위치

목업의 이모지는 위치를 설명하기 위한 임시 기호였다. 실제 게임에는 이모지를 사용하지 않는다. 캐릭터, 몬스터, 스킬 아이콘, 증강 아이콘, 배경은 통일된 색상표와 비율을 가진 AI 생성 픽셀 아트를 사용한다.

## 4. 핵심 게임 루프

1. 런 시작
2. 스테이지의 안전한 조건과 위험한 조건 중 하나 선택
3. 적 행동을 확인하며 턴제 전투
4. 승리 후 증강 3개 중 1개 선택
5. 생존 캐릭터 체력 회복 및 사망 상태 처리
6. 다음 스테이지로 진행
7. 1-5 보스 승리 시 챕터 클리어
8. 전멸 시 런 종료 및 런 능력 초기화
9. 플레이 결과에 따라 새 증강, 스킬 또는 몬스터 정보 해금

1-1은 고정 튜토리얼이며, 1-2부터 1-4까지 조건을 선택한다. 1-5는 고정 보스 조건이다.

### 안전한 조건

- 기본 난이도
- 일반 등급 중심의 증강 제시
- 안정적인 챕터 진행

### 위험한 조건

아래 조건 중 하나를 추가한다.

- 적 능력 강화
- 제한 턴 안에 승리
- 불리한 전장 상태

승리하면 희귀·전설 증강이 제시될 가능성이 높아진다.

## 5. 전투 시스템

### 라운드 순서

1. 살아 있는 적의 다음 행동과 대상 표시
2. 플레이어가 아군 세 명을 원하는 순서로 행동
3. 행동하지 못한 경우를 포함해 아군 턴 종료
4. 살아 있는 적이 예고한 행동 수행
5. 상태 효과와 쿨타임 갱신
6. 다음 라운드 시작

피해는 고정값이다. 기본 시스템에는 치명타, 회피, 무작위 피해 범위를 넣지 않는다. 전투 중 무작위성보다 플레이어의 선택이 결과를 설명할 수 있어야 한다.

### 위치

- 전장은 3개의 공격로로 구성한다.
- 각 진영의 공격로에는 전방과 후방 위치가 있다.
- 같은 열 공격, 관통, 보호, 밀치기, 범위 공격이 위치를 사용한다.
- 이동 규칙의 초기 가설은 캐릭터당 전투 중 1회 무료 위치 변경이다.
- 무료 이동이 전략보다 반복 조작을 늘리면 삭제하거나 턴당 무료 이동으로 변경한다.

이 이동 규칙은 확정 요구사항이 아니라 플레이테스트 항목이다.

### 스킬 자원

- 기본 공격: 항상 사용 가능
- 일반 스킬: 2~3턴의 개별 쿨타임
- 필살기: 공격, 피격 및 일부 효과로 충전되는 팀 공용 게이지 사용
- 한 번 충전된 게이지를 어느 캐릭터에게 사용할지가 주요 선택이다.

### 체력과 패배

- 세 캐릭터가 모두 쓰러지면 즉시 런 패배
- 한두 명이 쓰러져도 남은 캐릭터로 전투 계속
- 생존 캐릭터는 스테이지 종료 시 최대 체력의 25% 회복
- 쓰러진 캐릭터는 자동 부활하지 않음
- 다음 증강 보상을 포기하면 쓰러진 캐릭터 한 명을 최대 체력의 25%로 부활
- 여러 명이 쓰러졌다면 보상 포기 한 번에 한 명만 부활

## 6. 직업 설계

아래 수치는 첫 플레이테스트용이며 데이터 파일에서 쉽게 수정할 수 있어야 한다.

### 전사

- 역할: 보호, 피해 흡수, 위치 제어
- 최대 체력: 12
- 기본 공격력: 2
- 베기: 적 하나에 피해 2
- 방패 엄호: 지정 아군이 받을 다음 피해 중 최대 3을 대신 받음
- 방패 밀치기: 피해 2를 주고 가능하면 대상을 후방으로 이동
- 필살기 `철벽`: 다음 적 턴 동안 아군 전체가 받는 피해 대폭 감소

### 궁수

- 역할: 단일 화력, 관통, 처치 연쇄
- 최대 체력: 8
- 기본 공격력: 3
- 사격: 적 하나에 피해 3
- 관통 화살: 같은 공격로의 적 최대 둘에게 피해 2
- 표식: 대상이 받는 다음 피해 +2
- 필살기 `화살비`: 모든 적에게 피해 2

### 법사

- 역할: 광역 공격, 행동 지연, 배수 조건 생성
- 최대 체력: 7
- 기본 공격력: 2
- 마력탄: 적 하나에 피해 2
- 불꽃 폭발: 한 공격로의 모든 적에게 피해 2
- 빙결: 대상 행동을 다음 적 턴의 마지막으로 이동
- 필살기 `과충전`: 이번 아군 턴 동안 모든 아군 스킬 피해 ×2

## 7. 증강 시스템

전투 승리 후 증강 3개를 제시하고 하나를 선택한다.

### 등급

- 일반: 작은 수치 증가 또는 조건 보강
- 희귀: 스킬의 작동 방식 변경
- 전설: 곱연산, 재발동, 처치 연쇄 등 빌드의 핵심 효과

같은 증강을 다시 선택하면 2단계로 강화한다. 초기 프로토타입에서는 각 증강의 최대 단계를 2로 제한한다.

### 수치 원칙

- 기본 능력치와 일반 증강은 주로 덧셈 사용
- 희귀·전설 증강의 특정 조건에서만 곱셈 사용
- 동일 계열의 곱연산은 과도하게 겹치지 않도록 명시적인 적용 순서를 둠
- 플레이어에게 최종 피해의 계산 근거를 툴팁 또는 전투 로그로 표시

예: 기본 피해 `3 + 표식 2 = 5`, 과충전으로 `5 × 2 = 10`, 처치 재발사로 추가 공격.

### 초기 빌드 방향

1. 궁수 관통 및 처치 재발사
2. 법사 광역 폭발 및 재발동
3. 전사 보호 및 반격
4. 세 직업이 같은 공격로를 연속 공격하는 팀 연계

### 예시 증강

- `두꺼운 방패` 일반: 방패 엄호로 대신 받는 피해 -1. 2단계에서는 엄호 후 필살기 게이지 +1.
- `꿰뚫는 궤적` 희귀: 관통할 때마다 다음 대상 피해 +2. 2단계에서는 +3.
- `사냥 본능` 전설: 궁수가 적을 처치하면 가장 체력이 낮은 적에게 기본 공격 재발사.
- `불씨 전염` 희귀: 불꽃 폭발로 적이 쓰러질 때마다 같은 열에 피해 2 재발동.
- `완벽한 연계` 전설: 한 턴에 세 직업이 같은 열을 공격하면 마지막 공격 피해 ×2.
- `피의 가속` 전설: 아군이 한 명 이상 쓰러진 동안 모든 피해 ×2.

## 8. 챕터 1 구성

### 1-1 숲길의 습격

- 목적: 공격 대상, 적 행동 예고, 아군 행동 순서 학습
- 적: 쥐 2, 슬라임 1
- 난이도: 실패 가능성이 매우 낮음

### 1-2 좁은 오솔길

- 목적: 전방·후방, 밀치기, 같은 열 관통 학습
- 신규 적: 방패 고블린

### 1-3 매복조

- 목적: 표식, 광역 공격, 행동 지연 연계
- 신규 적: 궁수 고블린

### 1-4 정예 야영지

- 목적: 증원과 보호를 뚫으며 완성 중인 빌드 점검
- 적: 정예 대장과 부하

### 1-5 숲의 바리케이드

- 보스: 고블린 전차대장
- 두 단계 전투
- 부하 소환, 돌진 예고, 위치 압박 사용
- 플레이어가 밀치기와 관통을 이용해 보스 행동을 역이용하도록 설계

## 9. 영구 진행

런 중 획득한 전투 능력과 증강은 패배 시 모두 초기화한다.

영구 진행은 기본 능력치를 올리지 않고 선택지를 넓힌다.

- 새 증강 해금
- 새 스킬 변형 해금
- 발견한 몬스터 정보
- 이전 런의 통계와 기록

이를 통해 로그라이크의 긴장감을 유지하면서도 실패 후 다시 도전할 이유를 제공한다.

## 10. 기술 구조

### 저장소 구조

```text
/
  frontend/
  backend/
  docs/
  compose.yaml
  .env.example
  README.md
```

### 프론트엔드

- React
- TypeScript
- 게임 화면과 비공개 관리자 화면
- 전투 규칙은 가능한 한 UI와 분리된 순수 TypeScript 모듈
- 캐릭터, 적, 스킬, 증강, 스테이지는 데이터 중심으로 정의
- 브라우저 저장소에는 익명 기기 ID, 미전송 로그, 로컬 설정 저장

### 백엔드

- Spring Boot
- REST API
- 런 및 행동 이벤트 수집
- 집계와 필터 조회
- 관리자 데이터 조회
- 관리 API와 화면은 환경변수로 설정한 관리자 비밀값을 이용한 세션 보호
- 공개 배포 전에는 더 강한 인증 방식으로 교체

### 데이터베이스

- PostgreSQL
- Flyway 마이그레이션
- 게임 버전과 데이터 버전을 모든 런에 저장
- 원본 이벤트를 보존하고 분석용 집계는 재생성 가능하게 설계

### 실행 환경

- Git으로 형상 관리
- Gradle Wrapper 사용
- 운영체제별 절대 경로 금지
- Docker Compose로 전체 서비스를 실행할 수 있음
- 개발 중에는 PostgreSQL만 Docker로 실행하고 프론트엔드와 백엔드를 로컬 실행할 수 있음
- Windows와 macOS에서 같은 명령 체계를 최대한 유지

Docker는 필수가 아니라 재현 가능한 실행 수단이다.

## 11. 로그 데이터 설계

### 식별자

- `device_id`: 브라우저가 생성한 익명 UUID
- `run_id`: 한 번의 챕터 도전
- `battle_id`: 한 스테이지 전투
- `event_id`: 중복 전송 방지를 위한 고유 이벤트 ID
- `game_version`: 애플리케이션 버전
- `balance_version`: 캐릭터, 적, 증강 수치 버전
- `seed`: 적 패턴과 증강 제시 재현용 시드

### 원본 이벤트

- 런 시작 및 종료
- 경로 조건 제시 및 선택
- 전투 시작 및 종료
- 라운드 시작
- 적 행동 예고
- 캐릭터 행동, 스킬, 대상
- 피해, 치유, 보호, 상태 변화, 이동
- 캐릭터 및 적 사망
- 필살기 게이지 변화
- 증강 제시 목록
- 증강 선택 및 단계 강화
- 증강 포기와 캐릭터 부활
- 영구 콘텐츠 해금

클라이언트는 이벤트를 작은 묶음으로 비동기 전송한다. 네트워크 또는 서버 오류가 나도 전투를 중단하지 않는다. 미전송 이벤트는 브라우저에 저장하고 다음 기회에 재전송한다. 서버는 `event_id`로 중복을 제거한다.

## 12. 밸런스 연구실

관리자 화면은 비공개다. 로컬 프로토타입에서는 관리자 비밀값으로 보호하고 외부 네트워크에 공개하지 않는다.

### 원본 건수 우선

비율만 저장하거나 표시하지 않는다. 분석 화면은 다음 원본 건수를 우선 표시한다.

- 증강 제시 횟수
- 증강 선택 횟수
- 해당 증강 선택 후 각 스테이지 도달 횟수
- 해당 증강 선택 후 보스 도달 횟수
- 해당 증강 선택 후 클리어 횟수
- 해당 증강이 포함된 런의 총 피해 및 전투 턴 수

표시 예:

`제시 127회 / 선택 31회 / 보스 도달 18회 / 클리어 7회`

파생 지표는 원본 건수에서 계산한다.

- 선택률 = 선택 횟수 / 제시 횟수
- 보스 도달률 = 보스 도달 횟수 / 선택 횟수
- 선택 후 클리어율 = 클리어 횟수 / 선택 횟수
- 제시 기준 클리어 기여 관찰값 = 클리어 횟수 / 제시 횟수

대시보드는 모든 비율 옆에 분자와 분모를 함께 표시한다. 작은 표본은 별도 경고한다.

이 값들은 인과관계를 증명하지 않는다. 강한 플레이어가 특정 증강을 더 잘 선택하거나, 후반에 제시된 증강이 자연스럽게 높은 성과를 보이는 편향이 있을 수 있다. 스테이지, 위험 조건, 게임 버전, 함께 선택된 증강으로 필터링해 해석한다.

### 분석 기능

- 1-1 시작부터 보스 클리어까지 진행 퍼널
- 증강의 제시, 선택, 도달, 승리 원본 건수
- 증강 단독, 쌍, 최종 빌드 조합 비교
- 안전·위험 조건별 선택과 결과
- 캐릭터 및 스킬별 사용 횟수와 피해
- 특정 런의 턴별 재생
- 게임 버전과 밸런스 버전 필터
- CSV 내보내기
- `SELECT`만 허용하는 읽기 전용 분석 질의
- 질의 시간, 반환 행 수 및 접근 가능한 뷰 제한

### 향후 공개 가능 항목

원본 행동 로그는 공개하지 않는다. 익명 집계만 다음과 같은 재미 통계로 공개할 수 있다.

- 가장 많이 버려진 증강
- 한 턴 최고 피해
- 보스에게 가장 많이 전멸한 조합
- 가장 드문 클리어 빌드

## 13. 오류 처리

### 게임 클라이언트

- 잘못된 행동은 상태를 변경하기 전에 거부하고 이유 표시
- 전투 상태 전환은 단일 규칙 엔진을 통해 수행
- 저장 또는 로그 전송 실패가 플레이를 중단하지 않음
- 치명적인 상태 오류가 발생하면 현재 시드와 이벤트를 보존하고 안전하게 런 종료

### 서버

- 이벤트 입력 형식 검증
- 중복 이벤트 무시
- 존재하지 않는 런이나 전투 참조 거부
- 분석 API는 페이지 크기와 실행 시간을 제한
- 개인 정보나 자유 형식 사용자 입력은 수집하지 않음

- 이벤트 중복 저장 방지
- 분석 원본 건수와 파생 비율 계산

- PC 전체 런
- 모바일 가로 화면 전체 런
- 터치 대상 크기
- 적 행동과 피해 계산 근거 가독성

### 첫 프로토타입 완료 기준

- 1-1부터 1-5까지 10~15분 안에 완주 가능
- 승리, 패배, 재시작이 끊김 없이 동작
- 네 가지 빌드 방향 중 최소 둘 이상이 보스를 이길 수 있음
- 한 증강 조합만 모든 선택을 압도하지 않음
- 특정 런을 턴 단위로 재구성 가능
- 제시, 선택, 도달, 승리 횟수를 버전별로 비교 가능
- Windows와 macOS에서 개발 실행 가능
- 같은 Wi-Fi의 모바일 가로 화면에서 플레이 가능
